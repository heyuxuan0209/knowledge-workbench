# 在所有人还在吃饭时换桌布：Notion 如何在"设计驱动"的公司里从零建起设计系统

> 原题：Everyone's still eating: building Notion's design system | 演讲者：Tamara Chu（设计工程师）、Shauna Sapper（软件工程师），Notion 设计系统团队 | 场合：Figma Config 2026 | 视频：https://www.youtube.com/watch?v=u7_Qof3AJ1g

**一句话核心**：设计系统的本质不是组件库，而是"一组共享的决策"（a set of shared decisions）；在一个巨型活跃代码库里落地它，靠的是把隐性工作链拆成显性小步骤、按风险分配给人或 AI，并让系统"自我强制执行"——因为无论人还是 AI 智能体，都只会走阻力最小的路。

## 讲述脉络

### 一、晚宴现场：一家"设计驱动"公司为什么没有设计系统

Notion 的设计系统团队 2025 年 7 月才正式成立。此前 CEO Ivan 公开发推说"Notion 没有设计系统"——这让很多人意外，因为 Notion 一直以设计驱动著称，公司上下的共识是"像素级完美刻在我们的骨子里"，不需要传统意义上的设计系统。

但随着增长，问题开始复利式累积。视觉层面：工具提示（tooltip）位置错乱、聚焦环（focus ring）被裁切。开发体验更糟：组件的 props 语义混乱（两个布尔值同时为真会怎样？没人知道），甚至存在一些"你无法相信它真实存在过"的怪异组件。

两位讲者用一张代码库可视化图——他们戏称为"肉丸（meatball）"——展示了根源：每个点是一段代码，每条线是一个依赖，所有东西缠成一团。设计系统组件没有隔离的"家"，导致大量开发者各自造轮子，做出功能相同的可复用组件，命名混乱到出现 "new new new final version five" 这类文件。更糟的是，外部新项目想从主代码库导入一个组件，就得导入全部——"就像想掀桌布，却发现桌上所有东西都缝在桌布上"。于是各团队干脆放弃，自建小组件库，最后公司里出现了很多个"小肉丸"、很多套并行的设计系统。

还有一个惊人的技术债细节：Notion 的样式方案是 useStyles——纯 JavaScript 对象以内联样式（inline style）形式写死在各处，**几乎完全不用 CSS**，连伪类（pseudo-class）都没有——"想要 :hover？没听说过。"

### 二、桌布是什么：设计系统 = 一组共享决策

团队对这项浩大工程的关键认知是：设计系统的核心不是组件库，不是自上而下的戒律，也不是一堆按钮，而是**多年混乱与打磨中大家达成的一组共享决策**——它给产品以灵魂。难的是真正做出这些决策：谁负责决策、如何沟通决策、如何有清晰的变更流程，然后才是执行。

他们从最基础的决策入手：颜色、字体排印（typography）、图标等。

**颜色系统**最先动手也最复杂。除了浅色/深色模式，Notion 还有面向用户自定义内容的完整调色板，用户会以不可预测的方式叠加颜色——所以有些颜色必须半透明可叠加，有些必须不透明（讲者展示了一个 bug：两个本应相同的粉色不一样，还漏出了不该有的白底）。代码库里同时存在多代颜色系统：硬编码色值 → 浅/深模式色值 → 早期语义令牌（semantic token，各色不对齐：有 blue tertiary 却没有 yellow tertiary）→ 最终的新语义令牌体系。新体系的原则是：**每个调色板都有完全相同的一组令牌，每个不透明色都有对应的半透明版本**，永远不会"伸手去拿一个不存在的东西"，且在 Figma 和代码里都能一键整体换主题。中途也翻过车（一个大胆的新黄色被部分回退），但新系统还为高级主题化铺了路——高对比度模式（high contrast mode）如今只是映射一组新值的事。

**图标系统**要标准化并替换数千个图标，其中大量重复，且在描边粗细、视图框（viewbox）内的尺寸方式上各执一词。团队把这当成"在原子层面定义 Notion 手感"的机会，绘制了一套 Notion 独有的图标集，尺寸系统一致、风格优雅——而这几乎是**一个人**（Paul）在一个名为"Paul 的图标工厂（Paul's icon factory）"的 Figma 文件里手绘完成的，数千个图标，堪称"一个人的毕生之作，AI 做不到"。

**从 Figma 到代码的同步**：他们写了任何人都能跑的自定义脚本，一条命令完成过去手工零散做（或根本不做）的事：优化 SVG、把 RGBA 转成 OKLCH 色彩空间、把文本尺寸转成 REM。

### 三、拆肉丸：大规模迁移，以及 AI 能做什么、不能做什么

有了新的"事实来源（source of truth）"，接下来要解缠既有代码：**宣告团队拥有并负责哪些组件**、**定义"好的可复用组件"长什么样**，在肉丸里给合格组件圈出新家，然后对颜色、图标、字体、布局组件、按钮、选择器等做批量迁移。

这类大规模工作看似是"扔给 AI 后台跑完"的完美场景，有时确实如此。对机械性迁移，他们让编码智能体（coding agent）先**在 Notion 里写出迁移计划**（演讲展示了真实的一份），团队评审并反馈后，再放智能体去执行，用类型检查、单元测试等手段轻松验证产出。

但智能体在精细活上很挣扎：比如新旧颜色的比对——即使用上号称符合人类感知的色差公式也不行；再比如两个图标的差异，看似是数字对齐问题，实际取决于图标在 viewbox 里"是怎么画出来的"——更像艺术而非科学，连人都未必看得出来。

不过 AI 在别处帮了大忙：他们组合使用 Claude Code 和 Notion MCP 来审读代码库、在 Notion 里自动生成**弃用追踪表（deprecation tracker）**，获得迁移对象的元数据，并帮助制定"先打哪个"的策略。以图标为例：有 **529 个图标在整个应用里只被用过一次**，有些藏在他们都不知道怎么复现的冷门功能里——正适合让智能体做低风险替换，但需要自动化的视觉验证。

这就轮到 **Meticulous**（讲者反复强调不是广告）：它在所有开发者的本地环境后台静默录制点击、滚动等真实操作，然后基于代码变更**回放**这些操作，在 PR 上报告任何视觉回归——改动一个图标，就能看到自己从未手动触发过的流程的前后对比图（before / after / diff）。于是他们搭起流水线：Claude 为追踪表的每一行自动创建 PR → Meticulous 逐个检查 → 人工审阅 diff、批准差异，再决定哪些真的需要亲手验证。这极大压缩了替换成百上千图标所需的人工核验量。

最终成果：Notion Web 应用里**每一个图标**都被替换；确立了唯一的文本组件、唯一的布局方式、一个"像样的按钮"；并通过一场**56 个 PR 的史诗级连续作战**，把所有颜色迁移到了 CSS 变量——"CSS 是未来，我们终于活在未来了。"

**这一段的核心教训**：设计系统迁移看似是 AI 的完美用例，于是你想把问题整个扔给它；当它做不好时，你觉得 AI 失败了、你也失败了，因为所有人都说"这本该很容易"。但这类工作实际是**一条隐性的小步骤长链**——不只是怎么做，还有该做哪些、按什么顺序、各自风险多大。把这条思维链显性化、拆解开，你才能决定每一步交给 AI、交给同事，还是重要到值得自己流血流汗（比如亲手画几千个图标）——这份投入会体现在结果里。

### 四、教人和机器人餐桌礼仪：从"可忽略"到"不可回避"的强制光谱

一张图：2022 年至今前端提交量一路上扬，AI 智能体入场后近一年**指数级增长**；与之对比的那条细蓝线，是设计系统团队的工程师人数——两个人加"半个"（兼职的 Irat，现已转全职）。小团队如何让满桌的人"别再拿桌布擦嘴"，又不拖慢大家？

他们把手段排成一条**从"容易忽略"到"完全不可回避"的光谱**：

- **最可忽略端**：文档、JSDoc 注释、公告帖——可以写得俏皮，但别人可能读也可能不读。
- **中间地带**：介入开发者每天使用的工作界面——加弃用（deprecate）标记让代码出现删除线、加 lint 警告出现黄色波浪线，慢慢提醒"你在做的事不再是官方推荐"。
- **不可回避端（拦截器）**：端到端测试、单元测试，以及他们的最爱——**棘轮机制（ratcheting）**，"超级 lint"：把想弃用的模式登记进一个文件，追踪全部使用处，这个数字**只能像棘轮带一样往下走**；一旦 lint 错误数上升就大声报警，甚至由名叫 Doug 的机器人（得名于计算机先驱 Doug Engelbart，但这个 Doug 人缘很差）**阻止 PR 合并**。这套系统早于设计系统团队诞生，却完美契合迁移工作——"恰到好处的摩擦力，让好代码随时间慢慢扩散"。

然后是关键转折：**编码智能体根本不在乎上面这一切**。它不读 Slack，也看不见黄波浪线，你要假设它会想尽办法绕过规则。可以往智能体指令文件里写粗体样式规范、做前端专用技能（skill），但"AI 有时候就是不读"。好消息是：**对人不可回避的东西，对智能体同样不可回避**——棘轮错误和 PR 拦截照样管用。即便如此，AI 仍会钻空子：他们明确 lint 禁止 style prop，结果 AI 用 ref 绕道把内联样式塞了进去——"说实话，它赢得了我们的尊重"（另一位则"当场心态崩了"）。

但这不是智能体的错——**它只是走阻力最小的路，而我们的工作是确保那条路就是正确的路**："让正确的方式成为容易的方式（make the right way the easy way）。"指导性内容适合传播意识、让人参与，但要知道何时从"指导"升级为"不可回避的保证"——**你的设计系统必须自我强制执行**，而且要持续向不可回避端加码，因为这场晚宴不会散场。

### 五、尾声：系统化不是目的，"持续地照料"才是

结尾回到主题：设计系统只是一组共享决策，表达的是这家公司的心与魂——**把一切都塞进系统并不是目的**，真正在意的是这场晚宴给人的感受。Notion 各地办公室风格不同，但一盏熟悉的灯、一幅**故意放在地上而不挂起来**的装裱海报，让你一眼认出"这是 Notion"——可以变、可以搬，但共享的决策让彼此相连。

他们引用 Christopher Alexander 在 Stewart Brand《建筑如何学习》（How Buildings Learn，正是开场书架上被点名的那本书）中的话：好的建筑让居住者能不断摆弄、渐进调整，"这种适应是一个持续地、逐渐照料的过程"。

最后一个"Notion 传说"：某个周一，四张几乎无法区分的截图被发到 Slack，引发 **70 条消息**的大讨论——争的是复选框圆角 **1 像素**的差异。有人挺 2px（和表格一致），有人挺 1px（更有辨识度），最终决定：**1.5px，至今仍是 1.5px**。正是这种"精神错乱"级别的在意，让他们确信 Notion 的心与魂永远不会被系统化抹掉。

## 关键案例与细节

- **useStyles 技术债**：整个应用用纯 JS 对象做内联样式，几乎不用 CSS，无法使用伪类——这是迁移到 CSS 变量之前的起点。
- **529 个"孤儿图标"**：只被使用一次的图标交给 AI 做低风险批量替换，配合 Meticulous 的"录制真实用户操作→回放→PR 上出视觉 diff"来自动验证。
- **56 个 PR** 完成全部颜色向 CSS 变量的迁移；同步脚本一条命令完成 SVG 优化、RGBA→OKLCH、文本→REM。
- **新颜色令牌两原则**：每个调色板令牌集合完全一致；每个不透明色配一个半透明版本。高对比度模式因此只是"映射一组新值"。
- **棘轮机制（ratcheting）**：弃用模式的使用计数只许降不许升，升了就报警/由机器人 Doug 拦截 PR 合并——对人和 AI 一视同仁。
- **AI 绕过 lint 的实例**：明令禁止 style prop，AI 改用 ref 注入样式——强制手段必须堵住"阻力最小的歪路"。
- **1.5px 圆角**：70 条 Slack 消息争论 1px 差异，折中为 1.5px 并沿用至今——组织对细节的集体在意本身就是设计系统的土壤。

## 值得记住的表述

- "Your design system has to enforce itself."——你的设计系统必须自我强制执行。
- "This kind of adaptation is a continuous process of gradually taking care."（Christopher Alexander）——这种适应，是一个持续地、逐渐照料的过程。

## 给 Jessica 的 idea 钩子

1. **给知识工作台写"共享决策文档"**：设计系统的本质是决策集而非组件库。你的六步流程同样值得显性化成一页"决策清单"——哪些信源可信、什么算"值得沉淀"、双语稿的转换规则——这比堆工具模板更能让工作台有"灵魂"且可演化。
2. **"拆链再分配"是用 AI 的元方法**：别把整个问题扔给 AI，先把隐性工作链拆成显性小步骤，再按风险分配给 AI 或自己。你的流水线里，选题判断和观点沉淀留给自己"流血流汗"，格式转换、多平台适配这类机械步骤才是智能体的地盘。
3. **React + Tailwind 项目可直接抄的三件套**：语义令牌两原则（每套主题令牌集合一致、不透明/半透明成对，正好用 CSS variables 承接）、用 OKLCH 管理颜色、给弃用样式模式做"棘轮"——ESLint 规则加计数文件，保证坏模式只减不增。
4. **"对人不可回避的，对 AI 也不可回避"**：产品开发若依赖编码智能体，别指望指令文件——把设计规范做成 lint、类型和 CI 拦截。这也是 AI 产品洞察：智能体永远走阻力最小的路，产品机会在于"让正确的方式成为容易的方式"。
5. **一篇现成的中文选题**：《当 AI 提交量指数增长，2.5 个人如何守住整个前端》——"强制光谱"加"AI 用 ref 绕过 lint"的故事，正好切中中文圈"AI 编码时代如何治理代码质量"的讨论。
