# Fable 使用野外指南：解除束缚、找出未知、告别过去、拒绝合理

> 原题：Field Guide to Fable | 演讲者：Thariq Shihipar（Anthropic 技术团队成员，Claude Code 团队）| 场合：AI Engineer 大会 | 视频：https://www.youtube.com/watch?v=9fubhllmsBU

**一句话核心**：面对 Fable 这类能力"尖峰式"跃升的新一代模型，瓶颈已从模型转移到人——你需要解除自己施加给模型的束缚（少约束、多上下文）、系统性地挖出提示词里的"未知项"，并把野心调得"不合理"一点，因为构建变容易了，但创造价值依然很难。

## 讲述脉络

### 开场：Fable 是"地图打开了"

Thariq 在 Claude Code 团队工作。他宣布 Fable 当天晚些时候发布，并预告稍后与 Cat Wu、Simon Willison 的炉边谈话可能带来更新。他把 Fable 归入那类"你会一直记得的" Anthropic 模型——如 Sonnet 3.5 new、Opus 4、Opus 4.5。他的比喻：像 RPG 游戏走完新手教程后开放世界（open world）突然展开——可探索的东西极多，但也令人生畏和困惑。因此这场演讲是一份"野外指南"，分四部分：解除对 Claude 的束缚（unhobbling Claude）、找出你的未知（finding your unknowns）、处理失落感（dealing with the grief）、变得不合理（being unreasonable）。

### 第一部分：解除对 Claude 的束缚

核心论点：模型是"养育出来的，不是设计出来的"（grown, not designed）——Anthropic 不是设定"SWE-bench 要到 99%"再倒推实现，而是用数据、反馈和算力小心培育，边用边学。这意味着**真正限制模型的是我们**：我们给它的运行框架（harness）和提示词，本质上是我们对 Claude 理解程度的函数。"解除束缚"就是加深理解、释放能力。Fable 还很新，有大量理解空间待解锁。

他用一个反直觉的例子解释"能力悬垂"（capability overhang）：一条疯传的推文问，为什么 LLM 说不出哪些宝可梦的名字以 "aw" 结尾？一千只宝可梦里只有两只符合。普通聊天模型答不出——尽管它明明"知道"所有名字；但 Claude Code 可以：它会抓取全部宝可梦名单，写一个脚本过滤出结尾为 "aw" 的。这就是"尖峰式变聪明"：模型不是靠记忆加推理硬答，而是一旦获得代码执行工具，能力就跳跃式解锁。

接着他回顾了几次历史性的"尖峰"进步：

- **聊天模型 → Claude Code**：早期得手工粘贴代码库进上下文，天真的路线是"把上下文窗口做到一亿 token"；真正的突破是给模型"手臂"——bash 工具和操作环境的方式，让它自己构建、搜索上下文。这个洞察催生了 Claude Code。
- **Claude Tag（Slack 中的 Claude）**：解锁点是"主动工作"和"多人协作"——Claude Code 必须被提示才干活，而模型能"自己醒来干活"，这正在解锁新一波智能体（agent）。
- **系统提示词的演化**：团队最近**删掉了 Claude Code 系统提示词的 80%**。最佳实践随模型换代而变：Sonnet 3.5 new 时代是"小系统提示词 + 少量工具 + 大量示例"；模型变聪明后变成"大提示词 + 大量示例 + 很多工具"；而最新一代模型想要**更小的系统提示词**——示例反而会限制它，因为模型比你给的示例更有想象力。团队现在"给上下文而非约束"，尽量避免写 "do not do this" 这类禁令（旧模型却必需）。
- **AskUserQuestion 工具**：他刚进团队时做的功能，让 Claude 在规划时向用户弹出多选题。Opus 4 时代几乎调不动这个工具，要反复调教；到 Opus 4.5，他试着让模型"就这份规格问我 40 个问题"，模型开始能面试他；到 Opus 4.8 和 Fable，模型能直接生成一整份内嵌问题的 HTML 报告——交互方式彻底换代。
- **Markdown → HTML**：Markdown 起初是模型的好输出格式，能承载少量富信息；后来在 plan mode 里帮你预先理解 Claude 要做什么；如今 Claude 能直接产出深度 HTML 报告。

小结：这更接近生物学而非物理学——高度经验性、有机，规则未全知，但确有可积累的科学直觉。他推荐 Anthropic 的论文《On the Biology of a Large Language Model》（大语言模型的生物学），称其面向各种技术背景的读者。

### 第二部分：找出你的未知——也要"解除对自己的束缚"

框架："地图不是疆域"（the map is not the territory）。写代码时，你脑中的计划、提示词、规格是**地图**；实际代码库、真实世界、Claude 必须应对的约束是**疆域**。Claude 在疆域里撞见地图上没有的东西，就是一个"未知"（unknown）——一个你未指定的决策点。Fable 是第一个让他觉得**必须**认真清点未知项的模型：它跑得太远太广，会撞上大量未知。他直言："Fable 的瓶颈是我把地图和疆域对齐、找出未知的能力。"

他用一个 2×2 矩阵分类：**已知的已知**（写进提示词的需求）、**已知的未知**（知道自己还没想清楚的事）、**未知的已知**（太显然以至于不会写下来、"看到才认得"的偏好）、**未知的未知**（完全没考虑过、但一旦知道就会改变提示方式的事）。好消息是：可以用 Fable 本身来找未知。六个实操技巧：

1. **盲点扫描（blind spot pass）**：比如"我要在这个代码库接一个我一无所知的认证（auth）提供方，帮我做一次盲点扫描，找出相关的未知的未知，帮我把提示词写得更好"。Claude 可能翻认证模块发现反复出现的坑，也可以让它搜 git diff 或 Slack。用途很广，还能用来学新领域——他最近做视频剪辑时用它学调色（color grading）。他说模型几乎在所有领域都懂得比他多，关键是把知识"掏出来"。
2. **头脑风暴与原型**：对付"未知的已知"，尤其是设计这种"看到才认得"的东西。示例提示："我没有视觉品味，做一个包含四种风格迥异的设计方案的 HTML 页面，让我对着反应。"
3. **让 Claude 面试你**：方向定了但仍有大量未指定项时，让 Claude 采访你。要给它关于你、工作内容、所处阶段的上下文；加一句"优先问那些会改变架构的问题"效果极佳。
4. **给参考（references）**："给 Claude 一张地图的最好方式是给它另一张地图。"不写规格，直接丢一段代表你想要什么的代码——可以是另一个系统、另一种语言；做 React 组件时可以拿一个 HTML 原型当参考。Fable 在这方面表现惊人。
5. **实现笔记（implementation notes）**：让 Fable 运行中每遇到一个未知就记录下来，事后你能看到偏离发生在哪、为什么。
6. **让 Fable 考你**：完工后让它出题测验你发生了什么，确保你真正理解、能在提 PR 或合并时为这份工作负责。他强调"留在环路里"（staying in the loop）是与 Fable 协作最重要的事之一。

### 第三部分：处理失落感

第一次用 Fable 这个级别的模型，他同时感到巨大的"获得"与"失去"。LLM 之前的编程如今像"异国他乡"：他曾经营一家约 30 人的 YC 创业公司，因为写代码太难而被迫不断取舍——做性能优化还是做新功能原型？各要一两个月，只能二选一。几周前他回到那个老代码库，过去要几周的事几小时就做完了。"你怎么能不笑？但说实话，又怎么能不哭？"他深爱手写代码、爱"在脑中旋转代码库"的感觉，但也记得熬夜调试、连续数周毫无进展、"在失败里游泳"——他做过的项目大多失败了，大多数创业公司会破产。编程本质上极难，所以尽管怀念那些高光，他"回不去了"。他的结论："唯一的出路是穿过去"（the only way out is through）——继续认真学、留在环路里、解除模型的束缚，就能带着多得多的东西走到另一边。

### 第四部分：变得不合理

他最喜欢 Anthropic 文化的一点："我们相信取舍不是真的"（trade-offs are not real）。在前公司他习惯"讲道理"：列优先级、权衡、定季度重点。但如果全都做呢？**逼现实亲自把取舍摆到你面前**。Claude 和 Fable 的"数学"改变了取舍的计算方式——你脑中藏着无数隐式取舍，比如"好、快、便宜选两个"，现在是"三个全选"。做更有野心的工作的最好方式，是重新框定、让自己更有野心："证明智能体有用的唯一方式，是以前所未有的速度做出我们此生最好的工作。"例证：这份他很满意的演讲幻灯片，是前一晚用 Fable 约 4 小时做的。

他对在场的 AI 工程师说：世界正看着你们证明 AI 不是昙花一现，而是真能提高生产力、节省时间。他今年的决心是：更高产、但工作更少，把省下的时间留给在乎的人。最后的提醒：**构建变容易了，但创造价值依然很难**——AI 工程师容易沉迷于构建过程和工具配置，但目的是创造价值，而找到有价值的东西需要很多次挥棒尝试。结语：去探索，把它做成真的，少讲点道理。

## 关键案例与细节

- **宝可梦 "aw" 测试**：一千只宝可梦里两只名字以 aw 结尾；聊天模型答不出，Claude Code 靠"抓全量数据 + 写过滤脚本"答出——"能力悬垂"的最佳注脚。
- **删掉 80% 系统提示词**：Claude Code 团队为新一代模型大幅精简系统提示词；示例会限制比示例更有想象力的模型；用上下文替代禁令式约束。
- **AskUserQuestion 三级跳**：Opus 4 几乎调不动 → Opus 4.5 能就规格问 40 个问题做访谈 → Opus 4.8/Fable 直接生成内嵌问题的 HTML 报告。
- **提示词模板**："做一次盲点扫描，帮我找出未知的未知"；"我没有视觉品味，给我四个风格迥异的设计让我反应"；"优先问会改变架构的问题"。
- **4 小时做完大会演讲幻灯片**（前一晚，用 Fable）；老创业公司代码库里"几周的活变几小时"。

## 值得记住的表述

- "The models are grown, not designed."——模型是养育出来的，不是设计出来的。
- "Building is easier, but generating value is still hard."——构建变容易了，但创造价值依然很难。

## 给 Jessica 的 idea 钩子

1. **把"盲点扫描"做成工作台的固定环节**：在"信息搜寻→筛选"和动笔之间加一步——让 LLM 对选题做 blind spot pass，列出你的"未知的未知"。Thariq 用它学调色这种全新领域，正对应你写中英双线内容时快速进入陌生 AI 子领域的需求。
2. **用"让模型面试我"替代部分对话式探讨**：你的"探讨沉淀"环节可以借用他的模板——给模型你的背景、文章所处阶段，然后要求"优先问会改变文章结构/核心论点的问题"。用 40 个问题的访谈逼出你没写下来的隐性观点，正好服务"通过写作加深理解"的目标。
3. **"四个风格迥异的版本"解决选题与文风的'看到才认得'问题**：标题、开头、文章框架都可以让模型一次给出多个刻意拉开差异的版本供你反应——这是把"未知的已知"（说不出但认得出的品味）externalize 的低成本办法，也适合中英两个平台的差异化定位实验。
4. **产品启示——你的工作台提示词可能"约束过度"**：Claude Code 删掉 80% 系统提示词、"示例会限制想象力"、"给上下文而非禁令"这三条，值得直接拿来审计你工作台里筛选和生成环节的 prompt；同时注意这些最佳实践随模型换代会反转，提示词要按模型版本维护。
5. **"实现笔记 + 事后测验"移植到复盘环节**：让 LLM 在生成内容时记录每个"未指定决策点"，发布后再让它出题考你——既是复盘素材，也确保你能为 AI 参与的内容真正负责（他说的"留在环路里"），这本身也是一个可做成产品功能的交互模式。
