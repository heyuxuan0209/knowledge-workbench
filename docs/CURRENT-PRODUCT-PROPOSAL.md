## 一、产品愿景（Vision）

### 想解决的核心问题

使用者是一名 AI 领域的信息重度消费者，每天会看到 100+ 条 AI 相关资讯（来自 X、博客、GitHub、论文站等聚合渠道）。当前的痛点不是"信息不够"，而是：

1. **看完就忘**：读了很多，但没有沉淀，下次想找"上周看到那篇关于 XX 的文章"很困难
2. **缺乏深度**：单篇文章浏览是浅层消费，很少真正基于多篇内容做综合分析、产生自己的观点
3. **创作断层**：即使有了想法，从"读到的信息"到"产出一篇文章/一个产品构想"之间没有工具支撑，全靠自己脑内整理

### 想成为的产品

一个**从信息消费到知识产出的个人工作台**。不是资讯聚合器（那个问题已经被解决了），而是资讯之上的一层——帮助使用者把看到的内容转化为可追踪、可深度研究、可创作产出的资产。

### 明确不做的事

- 不做内容生产/采集本身（复用现成的聚合数据源）
- 不做社交/分享功能，这是纯个人工具
- 不追求"自动化帮用户决策"，工具应该放大用户自己的判断，而不是替代它

## 二、用户场景与 Workflow

以下是设想过的使用场景，**其中场景一有过初步验证，其余仍停留在设想阶段，没有验证其真实性和优先级**。

### 场景一：材料收集 + 深度对话分析

> 用户浏览资讯时看到几篇相关文章，想让 AI 基于这几篇内容一起做综合分析，而不是逐篇单独问一次。

设想的 workflow：
1. 浏览资讯列表
2. 把感兴趣的几篇标记/收集起来，形成一个"素材集合"
3. 针对这个素材集合发起对话，AI 的回答基于这些素材内容，而不是泛泛而谈
4. （未验证）产出的分析结果能被保存、复用、二次编辑

**开放问题**：素材收集的动作应该是什么形态？（拖拽只是众多候选交互中的一种，未必是最优解）"对话"是不是分析素材的最佳方式？是否应该有除了聊天以外的产出形式（比如结构化摘要、对比表格）？

### 场景二：主题/话题追踪（设想，未验证）

> 用户想按"话题"而不是"时间"来组织信息——比如"Agent 相关的讨论"这个话题，希望能看到它在过去一周/一月的演进，而不是淹没在时间线里。

**开放问题**：话题应该由系统自动识别聚类，还是用户手动创建维护？这两种模式对使用者的心智负担完全不同。"追踪"这个动作本身的价值在哪——是为了不错过更新，还是为了积累研究素材？

### 场景三：从信息到创作的桥接（设想，未验证）

> 用户读了一堆相关内容后，想把这些转化为自己的产出——一篇分析文章、一个产品构想、一份研究笔记。

**开放问题**：这个"桥接"应该发生在本产品内部（自带创作能力），还是本产品只负责"整理好素材，交给用户在别处创作"（比如导出、跳转到其他工具）？这两种定位对产品的复杂度和边界影响很大。

### 一个值得警惕的问题

以上三个场景目前是**并列列出**的，但它们可能并不是同一优先级、甚至可能不该被同一个产品同时满足。到底哪个场景是这个产品真正非做不可的核心，哪些只是"顺手也想要"，这个判断本身可能就需要被重新审视，而不是想当然接受"三个场景都重要"。

## 三、产品模块与架构（概念层面）

到目前为止，探索出的产品被想象成三个功能模块的组合：

1. **信息浏览层**：资讯列表 + 详情，纯消费型
2. **收集/组织层**：把散落的信息组织成某种"集合"（可能是主题、可能是单次分析的素材包）
3. **分析/产出层**：基于收集的内容，借助 LLM 做深度分析，产生某种输出

这三层目前被设计为**递进式**——用户依次经过浏览→收集→分析。但这个递进假设本身也值得质疑：

- 是不是所有内容都需要经过"收集"才能进入"分析"？有没有可能存在"边浏览边分析"的即时模式？
- "收集"和"分析"是否应该是同一个界面里的动作，还是分开的两个心智阶段？
- 三层里，哪一层是产品真正的差异化价值所在？如果只能做好一层，应该是哪一层？

### 界面组织的既有尝试（供参考，非定论）

目前尝试过的界面骨架，是把"浏览区"和"分析工作区"并排放在同一屏幕里（类似 IDE 或 Claude.ai 那种"内容 + 侧边工作区"的布局），设想是让用户不需要来回切换页面就能完成"看到内容→拉入分析"的动作。

但这个尝试目前还没有把"浏览"和"分析对话"两个动作真正统一到一次顺畅的体验里——具体是什么形式的割裂不重要，重要的是：**这暴露出一个更根本的问题：把"资讯流"和"深度工作区"塞进同一个屏幕，是不是本来就是一个有张力的设计目标？** 这两者对界面密度、心智模式的要求可能天然冲突（一个要"快速扫视很多条"，一个要"专注处理少数几条"），值得重新考虑这两个模式是否应该分离成完全不同的界面/使用时刻，而不是勉强拼在一起。

│──────────────│  │ 标题                │  │  （标题/摘要/来源/时间）│

**意图**：让"看到内容"和"收进工作区"在同一屏幕内一步完成，不用切页面。
**实际卡住的地方**：最右栏只有素材列表，没有对话输入框——能收集，但收集完之后要跑到另一个页面才能真正开始"分析"（见框线图 B）。这也是第三节提到的那个更根本的疑问的具体表现：把资讯流和工作区并排放，看似解决了"来回切换"，但只解决了"收集"这一半，"分析"那一半又被推到别处去了。

### 框线图 B：独立对话页（浏览与此处完全脱节）

```
┌────────────────────────────────────┬──────────────────────┐
│ ← 返回   对话标题        本次花费    │  Tab: 素材 / 产出       │
│─────────────────────────────────────│                       │
│                                     │  素材卡片列表           │
│         AI 回复（含引用素材内容）    │  （同框线图 A 的右栏，   │
│                              用户消息 │   在这里能正常复用）    │
│         AI 回复...                  │                       │
│─────────────────────────────────────│                       │
│ [输入框.............................] [发送]                  │
└────────────────────────────────────┴──────────────────────┘
```

**意图**：给"分析对话"一个专注、不被资讯流干扰的空间。
**实际卡住的地方**：这里能顺畅对话、素材也能正常显示引用，但因为资讯列表不在这个页面里，没法在对话过程中"临时想起再收一篇进来"，必须跳回框线图 A 重新走一遍收集流程再跳回来。

### 两张框线图放在一起说明的问题

这两张图分别把"收集"和"分析"做对了各自的一半，却从没有被拼成一次连续的体验——不是因为技术做不到，而是从最初设计这两张图的时候，就没有真正想清楚"收集"和"分析"到底应不应该发生在同一个空间、同一个时刻。这比"页面该怎么拼"是更前置的问题，值得评审者从头判断，而不是被这两张图的既有形态限制住。

### 如果需要交互形式上的启发（不代表应该照搬）

- Claude.ai / Cursor 的"内容 + AI 侧边栏"模式——但那些产品的"内容"通常是用户主动打开的单一文档，不是持续滚动的资讯流，这个差异可能是关键
- 邮件客户端的"列表 + 详情 + 收藏夹"三段式——但邮件是待处理的任务，资讯是待消费的内容，动机不同
- 读书笔记类工具（如 Readwise）的"高亮收集 + 定期回顾"模式——这个和"实时深度分析"的诉求可能更接近，值得参考

## 五、已经做过的重要设计决策（及其局限）

以下是过去探索中实际做过的选择，附上当时的理由——**但请注意，这些理由都是在没有真实用户验证的情况下做出的，本质上是猜测，不是已被证明的结论**。

### 决策：复用现成的资讯聚合数据源，不自建内容采集
**理由**：自建爬虫/翻译/筛选成本高，现成数据源已经解决了"信息获取"这个子问题。
**局限**：完全依赖对方的数据结构和更新节奏，产品的差异化空间被限定在"数据源之上能做什么"，如果数据源本身质量或覆盖面有问题，无法从根本解决。

### 决策：用 LLM 对话作为分析产出的主要形式
**理由**：对话是当前最容易实现、用户最熟悉的"和 AI 一起处理信息"的方式。
**局限**：这可能是"因为好实现"而不是"因为它是用户真正需要的形式"做出的选择。结构化的对比表格、时间线、思维导图等形式有没有可能对这个场景更合适，完全没有认真评估过。

### 决策：把"收集素材"设计成用户主动的、显式的动作（类似拖拽收藏）
**理由**：让用户明确控制"什么进入分析范围"，避免系统自动判断出错。
**局限**：这个假设是"用户愿意为每次分析做手动筹备工作"，但如果用户的真实习惯是"随手看到就想问一句"，显式收集这个步骤本身可能就是多余的摩擦，甚至是这个产品没人用起来的根本原因。

### 决策：暂不做用户账号体系、多设备同步、协作分享
**理由**：定位是个人工具，MVP 阶段降低复杂度。
**局限**：目前看没有明显局限，但如果产品方向发生大调整（比如决定往"团队共享的信息工作台"方向走），这个决策需要重新评估，不是一个"绝对正确"的设定。

## 六、给评审者的具体问题

比起"帮我优化这个方案"，更希望得到这些问题上的独立判断：

1. 三个场景（材料分析/主题追踪/创作桥接）是不是本该是三个不同的产品？如果必须收敛成一个，收敛的核心应该是什么？
2. "先收集素材、再分析"这个两步式的心智模型，是不是解决真实需求的最佳方式？有没有更贴近用户实际使用习惯的交互范式？
3. 这个产品的核心壁垒/差异化到底应该在哪一层——数据组织方式、AI 分析能力、还是创作产出的独特形式？如果说不清楚这一点，说明方向本身可能还没想清楚。
4. 如果完全抛开"资讯流"这个起点，重新想象"一个人如何把日常接触到的碎片信息变成有价值的产出"，会得到一个长得完全不一样的产品吗？
