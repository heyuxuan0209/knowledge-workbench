## ADR-001: 使用 AI HOT 作为数据源

**日期**: 2026-07-08  
**状态**: 🔶 已修订（被 ADR-008 补充：AI HOT 从"唯一数据源"改为"主渠道 + 分层策略"）  
**决策者**: 产品负责人

### 背景
需要稳定的 AI 资讯来源，要求：
- 中文友好（有翻译）
- 质量筛选机制
- 覆盖主流 AI 资讯渠道
- API 可访问

### 决策
使用 AI HOT (https://aihot.virxact.com) 的公开 API 作为唯一数据源

### 理由
✅ 已有高质量的中文翻译  
✅ 评分和筛选机制成熟（精选、评分）  
✅ 提供免费公开 API  
✅ 支持富文本正文（需详情 API）  
✅ 多源聚合（X、博客、GitHub、Reddit 等）  
✅ 日更新，数据新鲜

### 后果
- **依赖**: 依赖第三方服务稳定性
- **限制**: 需要遵循 API 使用限制（目前无明确限制）
- **数据新鲜度**: 取决于 AI HOT 更新频率（目前每日更新）
- **定制化困难**: 无法修改源数据的筛选逻辑

### 备选方案
| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|---------|
| 自建爬虫 | 完全可控 | 成本高，需维护，翻译困难 | 资源不足 |
| RSS 聚合 | 技术简单 | 无中文翻译，质量参差 | 体验差 |
| 购买第三方 API | 专业稳定 | 成本高 | 预算限制 |

## ADR-002: 浅色主题替代深色主题

**日期**: 2026-07-09  
**状态**: ✅ 已采纳  
**决策者**: 用户反馈驱动

### 背景
2026-07-08 MVP 采用深色主题（#1a1a1a），用户反馈：
- "太暗了，看久了眼睛累"
- "像个黑客工具，不像正经的工作台"
- "我更喜欢 Claude 官网那种浅色"

### 决策
改用浅色主题：米白背景（#fafaf9）+ stone 色系

### 理由
✅ **用户偏好**: 多数用户习惯浅色界面  
✅ **专业感**: 企业级产品多用浅色  
✅ **阅读舒适**: 长时间阅读更友好  
✅ **主流趋势**: Claude、Linear、Notion 都是浅色为主

### 后果
- **设计成本**: 需要重新设计所有组件的配色
- **对比度调整**: 浅色背景下文字对比度需要仔细调整
- **可访问性**: 更容易满足 WCAG 对比度要求

### 保留深色模式？
**暂不保留**，理由：
- MVP 阶段避免增加复杂度
- 待用户量增长后根据反馈决定
- 如需实现，成本约 2 天工作量

## ADR-003: 使用 SVG 图标替代 Emoji

**日期**: 2026-07-09  
**状态**: ✅ 已采纳  
**决策者**: 设计驱动

### 背景
原设计使用 Emoji（📥📌🔬🗺️）作为图标，用户反馈：
- "看起来不够专业"
- "像玩具或游戏界面"
- "不同系统显示的 emoji 样式不一致"

### 决策
使用 Heroicons 的 SVG 图标，内联到组件中

### 理由
✅ **专业统一**: 所有平台显示一致  
✅ **无依赖**: 内联 SVG，不需要额外库  
✅ **可定制**: 可修改颜色、大小  
✅ **Tailwind 官方**: 与 Tailwind CSS 完美集成

### 图标库选择对比
| 库 | 优点 | 缺点 | 结果 |
|----|------|------|------|
| Heroicons | Tailwind 官方，简洁 | 图标数量较少 (~300) | ✅ 采纳 |
| Lucide | 设计精美，数量多 (~1000) | 需额外安装 | 备选 |
| FontAwesome | 图标最多 (~7000) | 体积大，风格重 | 不适合 |
| Material Icons | Google 官方 | 风格偏 Android | 不适合 |

### 后果
- **迁移成本**: 需要逐个替换所有 emoji
- **代码量增加**: 每个图标 ~10 行 SVG 代码
- **性能影响**: 可忽略（内联 SVG 比图片快）

## ADR-004: 三栏对话式布局

**日期**: 2026-07-09  
**状态**: ❌ 已推翻（被 ADR-011 取代：资讯流与深度工作区强塞一屏存在心智张力，改为按使用模式分离）  
**决策者**: 产品设计

### 背景
产品定位是"工作台"而非"新闻阅读器"，用户需要：
- 快速浏览信息流
- 深度研究某个主题
- 与 LLM 对话分析内容

### 决策
采用三栏固定式布局：
- 左侧（280px）：导航 + 工作区列表
- 中间（flex）：内容主区（列表/详情/对话）
- 右侧（400px）：辅助面板（可折叠）

### 参考对象
- **Claude.ai**: 三栏对话式工作台
- **Cursor**: 左侧文件树 + 中间编辑器 + 右侧 AI
- **Linear**: 左侧导航 + 中间 issue 详情 + 右侧面板

### 理由
✅ **信息密度高**: 一屏显示更多内容  
✅ **符合工作习惯**: 程序员/知识工作者熟悉此布局  
✅ **支持多任务**: 左侧切换，中间工作，右侧参考  
✅ **可扩展**: 右侧面板可根据场景变化

### 后果
- **移动端不友好**: 需要单独设计移动端布局
- **最小屏幕要求**: 建议 1280px 以上
- **复杂度增加**: 需要管理三个区域的状态同步

### 响应式策略
- `>= 1280px`: 三栏布局
- `768px - 1279px`: 双栏（隐藏右侧面板）
- `< 768px`: 单栏（堆叠布局）

**当前状态**: 仅实现桌面端，移动端待优化

## ADR-005: Node.js 内置 sqlite 替代 better-sqlite3

**日期**: 2026-07-08  
**状态**: ✅ 已采纳  
**决策者**: 技术限制

### 背景
原计划使用 `better-sqlite3` 作为数据库驱动，但在 Node.js 26 上编译失败：
```
error: node-gyp rebuild failed
```

### 决策
使用 Node.js 26 内置的 `node:sqlite` 模块

### 理由
✅ **无需编译**: 内置模块，开箱即用  
✅ **性能足够**: 对于本地数据库场景性能充足  
✅ **官方支持**: Node.js 官方维护  
✅ **零依赖**: 不增加 node_modules 体积

### 后果
- **API 不同**: 需要调整数据库操作代码
- **生态较小**: 社区资源和示例较少
- **功能限制**: 部分高级特性不支持（如 WAL 模式）

### 性能对比
| 指标 | better-sqlite3 | node:sqlite |
|------|---------------|------------|
| 同步查询 | ✅ 支持 | ✅ 支持 |
| 异步查询 | ✅ 支持 | ✅ 支持 |
| 事务 | ✅ 完整支持 | ✅ 基础支持 |
| 性能 | 极快 | 快 |

**结论**: 对于本项目（< 10万条数据）性能差异可忽略

## ADR-006: Source 作为轻量身份层，而非独立抓取系统

**日期**: 2026-07-10  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人 + 架构评审

### 背景
产品的真实工作流是"从内容发现感兴趣的人 → 关注 → 未来信息被这些人重新加权"，心智接近 X 的推荐机制。评审曾倾向把 Source 设计成平行的抓取子系统，被判定为过度设计。

### 决策
Source 是**架在现有内容摄入之上的一层身份识别 + 标记**，不在 Content 之外重建抓取体系。一个 Source 可关联多个 Platform（X / YouTube / GitHub / Newsletter / 公众号）。

### 理由
✅ 符合"先看内容再认人"的真实习惯（对标 X 推荐机制）  
✅ 轻量，不引入重基础设施  
✅ 支撑三个价值点：feed 过滤视角、Topic 演进的"谁说的"、创作引用可信度

### Source 进入系统的三条路径（且只有这三条）
1. **涌现关注**：AI HOT 刷到内容 → 感兴趣 → 关注这个人
2. **批量导入**：已关注的一批账号一次性建档
3. **一次性丢链接**：只翻译解读、用完即走（等同于 ADR-009 的即兴分析，不建立持续关系）

### 后果
- 需要维护身份归一（同一人多平台账号的合并）
- 不承担独立抓取系统的复杂度

## ADR-007: 主动追踪的成本分层（硬约束）

**日期**: 2026-07-10  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人

### 背景
"关注一个人"这个动作，对不同平台的实现成本完全不同，不能一视同仁。

### 决策
按关注对象所在平台分三层，`trackMode` 由平台决定而非用户随意配置：

| 关注对象所在平台 | 追踪方式 | trackMode | 成本 |
|-----------------|---------|-----------|------|
| 在 AI HOT 覆盖范围内 | 纯被动，等 AI HOT 推送 | passive | 零额外成本 |
| 不在 AI HOT，但在 X/YouTube | 产品主动查询，接官方 API/RSS | active | 有实现与维护成本 |
| 微信公众号 | 不主动抓取，被动等推送 + 跳转原文 | passive（恒定） | 不承担逆向抓取风险 |

### 理由
✅ 避免"关注"功能给出无法兑现的更新承诺  
✅ 公众号无公开 API，主动抓取有风险和维护成本，明确不做  
✅ UI 需体现 passive / active 差异

### 后果
- 关注 UI 要标明该 Source 是被动等待还是主动追踪
- 公众号 Source 明确不承诺主动更新

## ADR-008: 数据源从"唯一"改为"主渠道 + 分层"

**日期**: 2026-07-10  
**状态**: ✅ 已采纳（修订 ADR-001）  
**决策者**: 产品负责人

### 背景
ADR-001 将 AI HOT 定为唯一数据源。真实工作流需要跟踪 AI HOT 未覆盖的特定关注对象。

### 决策
- **AI HOT**：主渠道，发现 + 被动推送主力（保留 ADR-001 的复用判断）
- **微信公众号**：不抓取，只被动等 AI HOT + 跳转原文
- **X / YouTube（AI HOT 未覆盖的关注对象）**：接官方 API / RSS 主动查询
- **RSS / 博客**：RSSHub 转 RSS + 标准解析

### 理由
✅ 采集层不是壁垒，全部复用  
✅ 差异化在其上的身份层 + Topic 演进 + 创作桥接  
✅ 避免单一数据源的覆盖盲区

## ADR-009: 即兴分析取代显式素材收集

**日期**: 2026-07-10  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人 + 架构评审

### 背景
旧方案设计了"拖拽收集素材 → 再分析"的两步式流程，假设用户愿意为每次分析做手动筹备。真实习惯是"随手看到就想问一句"，显式收集是多余摩擦。

### 决策
- 去掉工作区、拖拽收集
- 即兴分析 = 选中 feed 内容 或 粘贴外部链接 → 弹窗对话（类 NotebookLM/ChatGPT）→ 有价值可一键 Save
- 选中已有内容 vs 粘贴外部链接是同一动作的两种触发，不拆成两个功能

### 理由
✅ 从"看到内容"到"开始对话" < 10 秒  
✅ 消除没人用起来的根本摩擦  
✅ 对话产出可选沉淀，无价值则用完即走

### 后果
- 需支持键盘完成多选 + 唤起对话 + 发送

## ADR-010: 多语言摄入流水线作为一等公民

**日期**: 2026-07-10  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人

### 背景
旧方案未处理多语言。产品核心是把英文内容转为中文知识表达，这不是"翻译标题摘要"的轻量层能覆盖的。

### 决策
在设计内容模型时就把多语言作为一等公民，Phase 1 必做。完整链路：

```
YouTube → Transcript（有字幕取字幕，无则 Whisper 转录）
       → 中文翻译（术语表保一致；分级：重要内容 LLM，一般 NMT）
       → 章节分段（LLM 重新分章）
       → Builder 观点提取
       → Topic 归类
```

文章/论文走简化版（无 transcript 步骤）。

### 关键决策
- 标题/摘要**必须**翻译（供 feed 扫描）
- 全文可选翻译（成本/质量分级）
- **原文始终保留**（精读对照）
- 术语表跨内容保证 AI/Agent/RAG 等术语一致

### 复用
youtube-transcript-api · Whisper API · DeepL（NMT 主力）· LLM（重要内容 + 术语修正 + 分段 + 观点提取）

## ADR-011: 按使用模式分离界面，不强塞一屏

**日期**: 2026-07-10  
**状态**: ✅ 已采纳（取代 ADR-004）  
**决策者**: 架构评审

### 背景
ADR-004 把资讯流和深度工作区并排放同一屏。评审发现二者是两种冲突的心智模式：资讯流要"快速扫视很多条"，工作区要"专注处理少数几条"，对界面密度和认知负荷要求天然冲突。

### 决策
按三种使用模式分离，不勉强拼在一屏：
- **Mode 1 即兴分析（Ephemeral）**：轻、快、用完即走
- **Mode 2 主题研究（Persistent）**：慢、深、长期演进；界面独立于 feed
- **Mode 3 创作桥接（Output）**：基于成熟 Topic 产出

三种模式共享同一份数据（Content / Source / Topic），但入口与界面分离。

### 理由
✅ 消除"扫视"与"专注"的界面张力  
✅ 每种模式可独立打磨到位

## ADR-012: 不做 Insight Mining（主动洞察挖掘）

**日期**: 2026-07-10  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人

### 背景
评审曾把"产品机会 / 趋势判断"等洞察挖掘列为核心缺失能力，建议系统主动挖掘并推送。

### 决策
**不做**独立的 Insight Mining 模块。产品机会、趋势判断等洞察，是 Mode 3 创作桥接中人与 agent 对话讨论的**自然产出**，不是系统主动挖掘、主动推送的功能。

### 理由
✅ 该功能无真实需求验证支撑  
✅ 主动挖掘推送易在没人验证的情况下消耗大量设计精力  
✅ 洞察作为对话自然产出已能满足需求

### 后果
- 从缺失能力清单移除
- 若未来 Mode 3 做实后有需要，再重新评估

## Phase 演进路径（修订后）

| Phase | 内容 |
|-------|------|
| **Phase 1** | Mode 1 即兴分析（ADR-009）+ 多语言摄入流水线（ADR-010）+ Source 轻量识别标记（ADR-006） |

> Source 识别标记从原计划的独立 Phase 2 提前到 Phase 1，与即兴分析同期。它是内容摄入之上的一层标记，不是要新建的复杂子系统。
