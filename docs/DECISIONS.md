## ADR-001: 使用 AI HOT 作为数据源

**日期**: 2026-07-08  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人

### 背景
需要稳定的 AI 资讯来源，要求：
- 中文友好（有翻译）
- 质量筛选机制
- 覆盖主流 AI 资讯渠道
- API 可访问

### 决策
使用 AI HOT (https://aihot.virxact.com) 的公开 API 作为唯一数据源

### 理由
✅ 已有高质量的中文翻译  
✅ 评分和筛选机制成熟（精选、评分）  
✅ 提供免费公开 API  
✅ 支持富文本正文（需详情 API）  
✅ 多源聚合（X、博客、GitHub、Reddit 等）  
✅ 日更新，数据新鲜

### 后果
- **依赖**: 依赖第三方服务稳定性
- **限制**: 需要遵循 API 使用限制（目前无明确限制）
- **数据新鲜度**: 取决于 AI HOT 更新频率（目前每日更新）
- **定制化困难**: 无法修改源数据的筛选逻辑

### 备选方案
| 方案 | 优点 | 缺点 | 为何放弃 |
|------|------|------|---------|
| 自建爬虫 | 完全可控 | 成本高，需维护，翻译困难 | 资源不足 |
| RSS 聚合 | 技术简单 | 无中文翻译，质量参差 | 体验差 |
| 购买第三方 API | 专业稳定 | 成本高 | 预算限制 |

## ADR-002: 浅色主题替代深色主题

**日期**: 2026-07-09  
**状态**: ✅ 已采纳  
**决策者**: 用户反馈驱动

### 背景
2026-07-08 MVP 采用深色主题（#1a1a1a），用户反馈：
- "太暗了，看久了眼睛累"
- "像个黑客工具，不像正经的工作台"
- "我更喜欢 Claude 官网那种浅色"

### 决策
改用浅色主题：米白背景（#fafaf9）+ stone 色系

### 理由
✅ **用户偏好**: 多数用户习惯浅色界面  
✅ **专业感**: 企业级产品多用浅色  
✅ **阅读舒适**: 长时间阅读更友好  
✅ **主流趋势**: Claude、Linear、Notion 都是浅色为主

### 后果
- **设计成本**: 需要重新设计所有组件的配色
- **对比度调整**: 浅色背景下文字对比度需要仔细调整
- **可访问性**: 更容易满足 WCAG 对比度要求

### 保留深色模式？
**暂不保留**，理由：
- MVP 阶段避免增加复杂度
- 待用户量增长后根据反馈决定
- 如需实现，成本约 2 天工作量

## ADR-003: 使用 SVG 图标替代 Emoji

**日期**: 2026-07-09  
**状态**: ✅ 已采纳  
**决策者**: 设计驱动

### 背景
原设计使用 Emoji（📥📌🔬🗺️）作为图标，用户反馈：
- "看起来不够专业"
- "像玩具或游戏界面"
- "不同系统显示的 emoji 样式不一致"

### 决策
使用 Heroicons 的 SVG 图标，内联到组件中

### 理由
✅ **专业统一**: 所有平台显示一致  
✅ **无依赖**: 内联 SVG，不需要额外库  
✅ **可定制**: 可修改颜色、大小  
✅ **Tailwind 官方**: 与 Tailwind CSS 完美集成

### 图标库选择对比
| 库 | 优点 | 缺点 | 结果 |
|----|------|------|------|
| Heroicons | Tailwind 官方，简洁 | 图标数量较少 (~300) | ✅ 采纳 |
| Lucide | 设计精美，数量多 (~1000) | 需额外安装 | 备选 |
| FontAwesome | 图标最多 (~7000) | 体积大，风格重 | 不适合 |
| Material Icons | Google 官方 | 风格偏 Android | 不适合 |

### 后果
- **迁移成本**: 需要逐个替换所有 emoji
- **代码量增加**: 每个图标 ~10 行 SVG 代码
- **性能影响**: 可忽略（内联 SVG 比图片快）

## ADR-004: 三栏对话式布局

**日期**: 2026-07-09  
**状态**: ✅ 已采纳  
**决策者**: 产品设计

### 背景
产品定位是"工作台"而非"新闻阅读器"，用户需要：
- 快速浏览信息流
- 深度研究某个主题
- 与 LLM 对话分析内容

### 决策
采用三栏固定式布局：
- 左侧（280px）：导航 + 工作区列表
- 中间（flex）：内容主区（列表/详情/对话）
- 右侧（400px）：辅助面板（可折叠）

### 参考对象
- **Claude.ai**: 三栏对话式工作台
- **Cursor**: 左侧文件树 + 中间编辑器 + 右侧 AI
- **Linear**: 左侧导航 + 中间 issue 详情 + 右侧面板

### 理由
✅ **信息密度高**: 一屏显示更多内容  
✅ **符合工作习惯**: 程序员/知识工作者熟悉此布局  
✅ **支持多任务**: 左侧切换，中间工作，右侧参考  
✅ **可扩展**: 右侧面板可根据场景变化

### 后果
- **移动端不友好**: 需要单独设计移动端布局
- **最小屏幕要求**: 建议 1280px 以上
- **复杂度增加**: 需要管理三个区域的状态同步

### 响应式策略
- `>= 1280px`: 三栏布局
- `768px - 1279px`: 双栏（隐藏右侧面板）
- `< 768px`: 单栏（堆叠布局）

**当前状态**: 仅实现桌面端，移动端待优化

## ADR-005: Node.js 内置 sqlite 替代 better-sqlite3

**日期**: 2026-07-08  
**状态**: ✅ 已采纳  
**决策者**: 技术限制

### 背景
原计划使用 `better-sqlite3` 作为数据库驱动，但在 Node.js 26 上编译失败：
```
error: node-gyp rebuild failed
```

### 决策
使用 Node.js 26 内置的 `node:sqlite` 模块

### 理由
✅ **无需编译**: 内置模块，开箱即用  
✅ **性能足够**: 对于本地数据库场景性能充足  
✅ **官方支持**: Node.js 官方维护  
✅ **零依赖**: 不增加 node_modules 体积

### 后果
- **API 不同**: 需要调整数据库操作代码
- **生态较小**: 社区资源和示例较少
- **功能限制**: 部分高级特性不支持（如 WAL 模式）

### 性能对比
| 指标 | better-sqlite3 | node:sqlite |
|------|---------------|------------|
| 同步查询 | ✅ 支持 | ✅ 支持 |
| 异步查询 | ✅ 支持 | ✅ 支持 |
| 事务 | ✅ 完整支持 | ✅ 基础支持 |
| 性能 | 极快 | 快 |

**结论**: 对于本项目（< 10万条数据）性能差异可忽略

## ADR-006: 六阶段流水线架构（面向未来 Agent/插件化）

**日期**: 2026-07-14  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人 + V2 设计讨论  
**详细设计**: `docs/PRODUCT-DESIGN-V2.md`

### 背景
产品定位为一人媒体公司的 AI 流水线（信息输入 → 理解 → 观点 → 创作 → 分发）。第一版是个人工作台，但长期可能演化为多 Agent 流水线（热点发现 → 选题 → 深研 → 生成 → 发布）或可插拔插件市场。

### 决策
产品骨架为六阶段：**采集 → 理解 → 沉淀 → 洞察 → 创作 → 发布**。架构约束：
1. 每阶段是边界清晰的模块（独立 service + 明确输入输出契约）
2. 模块间只通过数据层和 HTTP API 通信，不互相 import
3. LLM 调用收敛到统一 provider 层（当前 Deepseek，可替换）
4. 定时任务做成独立可执行脚本（sync-*.js 惯例），未来可升格为 Agent

### 后果
- 第一版仍是单体，但拆分线已预埋
- 新增功能必须归属到某个阶段模块，禁止跨模块耦合

## ADR-007: Source = 优质源登记处（track_mode 四档分层）

**日期**: 2026-07-14（修订 2026-07-10 的"纯被动标记"决策）  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人

### 背景
用户核心诉求：发现好的 builder → 登记为信息源 → 持续追踪。原决策（Source 仅为被动标记层）无法满足；但也明确**不做订阅系统**（重建订阅推送是巨坑）。

### 决策
Source 是"优质源登记处"：丢入公众号名称 / X 链接 / 网页链接 → 自动识别身份 → 登记。登记后的效果**只有两个**：该源内容进 Feed + 高权重排序。

| 情况 | track_mode | 行为 |
|------|-----------|------|
| AI HOT 已覆盖 | passive | 纯标记 + 加权，零成本 |
| 有 RSS（博客/Newsletter/播客） | active-rss | RSS 轮询 |
| X/YouTube 且 AI HOT 未覆盖 | active-query | 轻量定期查询（优先 skills 管道，浏览器抓取兜底） |
| 微信公众号 | link-only | 标记 + 等推送 + 跳转原文，不抓取 |

### 后果
- 飞轮闭环：内容详情页提供"把作者加为信息源"一键操作
- 公众号维持不抓取（单篇也只跳转原文 + 用已有摘要）

## ADR-008: 洞察边界：自有信息流聚合 + 节奏化简报，不做全网挖掘

**日期**: 2026-07-14（重新划定 2026-07-13 "不做 Insight Mining" 的边界）  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人

### 决策
- ✅ **做**：基于用户自己信息流的 Story 聚类 → **选题建议**（角度 + 为什么是现在 + 支撑素材 + 共识/非共识），是核心功能而非附属
- ✅ **做**：节奏化简报——日报（今日焦点 + 2-3 个选题）/ 周报（本周动向）/ 月报（趋势），**不实时刷新**
- ❌ **不做**：全网无差别趋势挖掘、"系统告诉你该研究什么"

### 后果
- Feed 页顶部改为日报（今日焦点 + 选题），Feed 卡片复用 AI HOT 全部标签 + 摘要 + 全文解读入口 + 跳转原文
- 新增 Idea（选题）、Report（简报）实体

## ADR-009: 活知识库：Topic 升级为 AI 维护的活页（Karpathy LLM-wiki 式）

**日期**: 2026-07-14  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人

### 背景
用户对"死知识变活知识"的定义：类似 Karpathy LLM wiki 的**自生长知识库**——AI 通过已有内容不断涌现新内容，而非静态归档。

### 决策
- Topic 重新定义为**活页（Living Page）**：AI 维护的活文档（认知综述 + 各方观点 + 共识/非共识），不是文件夹
- **同化机制**：新素材被保存或命中已追踪 Topic 时，AI 并入活页——补证据、修正论断、标记矛盾，写入 changelog
- **changelog 即演进时间线**：演进阶段从修订历史自然涌现，不手动维护
- **涌现**：周报/月报生成时回顾同化记录，提出新活页建议、跨页关联、矛盾预警、新选题

### 成本控制
同化只作用于用户保存的素材和已追踪 Topic 命中的内容，不对全量 Feed 执行。"用户选择保存"即质量门槛。

## ADR-010: 沉淀采用 NotebookLM 模式（对话不落库，选择性保存素材卡片）

**日期**: 2026-07-14  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人

### 决策
- 即兴对话维持无状态（轻、快、用完即走），**不默认保存**
- 用户选择"保存到笔记" → 生成**素材卡片（Note）**：结构化摘录 + 来源引用（回链 Content，创作时可溯源）
- 立场批注（同意/反对/存疑）：字段预留，**暂不做 UI**（见 TBD-004）
- 解读产物升级为**结构化解读**：核心论点 / 论据案例 / 金句 / 与已有观点异同；英文内容直接产出中文结构化版

## ADR-011: 创作输出按平台分化模板

**日期**: 2026-07-14  
**状态**: ✅ 已采纳  
**决策者**: 产品负责人

### 决策
同一素材集（Notes + 活页）按平台分化输出：
- **X thread / 短图文**：钩子 + 分条，可从单篇解读直接快速生成（M2 先行）
- **公众号长文**：大纲 → 草稿，深度综述
- **口播视频脚本**：开场钩子、口语化、时长约束（抖音/视频号）

每段可溯源到素材卡片 → 原始 Content。第一版发布只做导出/复制，不做自动发布。
