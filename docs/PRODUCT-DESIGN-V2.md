# Knowledge Workbench 产品设计 V2

**日期**: 2026-07-14
**状态**: 与产品负责人讨论后确定的设计基线，取代 HANDOFF-2026-07-13 中的部分决策（修订清单见第八节）
**读者**: 后续所有参与开发的 Agent / 协作者

---

## 一、第一性原理与定位

用户是**一人媒体公司**（独立开发者 / AI 产品经理 / 自媒体人），价值链为：

> 信息输入 → 理解 → 形成观点 → 生产内容 → 多平台分发 → 积累个人 IP 影响力

本产品是给这条价值链装上 AI 的流水线。它不是阅读器（终点不是"看完"），不是笔记工具（终点不是"存下"），**终点是"产出带用户观点的内容"**。所有功能设计向这个终点收敛。

**内容飞轮**：

```
发现（Feed/热点） → 解读（结构化理解） → 沉淀（素材 + 活知识库）
   ↑                                          ↓
追踪加权（Source 反哺 Feed 排序） ← 创作（日报选题/Topic 综述 → 多平台产出）
```

## 二、产品骨架：六阶段流水线

```
① 采集 → ② 理解 → ③ 沉淀 → ④ 洞察 → ⑤ 创作 → ⑥ 发布
Acquire   Understand  Retain    Synthesize  Create     Publish
```

**面向未来的架构约束**（重要）：第一版是单人使用的单体工作台，但产品的长期形态可能是多 Agent 流水线（热点发现 → 爆款选题 → 深度研究 → 内容生成 → 内容发布）或可插拔的 skills/插件市场。因此：

1. 每个阶段是一个**边界清晰的模块**：独立 service + 明确的输入输出契约。
2. 模块之间**只通过数据层和 HTTP API 通信，不互相 import**。
3. LLM 调用收敛到统一 provider 层（当前 Deepseek，可替换）。
4. 定时任务（同步、日报、周报）做成独立可执行脚本（延续 sync-*.js 惯例），未来可直接升格为 Agent。

## 三、各阶段设计

### ① 采集（Acquire）

**原则：能复用现有的绝不重复造轮子。**

| 数据源 | 方式 | 状态 |
|--------|------|------|
| AI HOT | 主干管道，已监测 X 等平台的优质 builders | ✅ 已接入，复用 |
| follow-builders / ai-insight-hub skills | 作为现成数据管道接入采集层（X / YouTube builders 内容） | 🆕 新增适配器 |
| HackerNews | 官方 API | ✅ 已接入 |
| RSS（博客 / Newsletter / 播客 shownotes） | rss-parser | ✅ 已接入 |
| GitHub Trending | AI 相关热门仓库，覆盖"从产品/开源汲取思路"需求 | 🆕 低成本接入 |
| 万能收口 | 任意链接（网页/YouTube/X 帖/公众号文章）或公众号名称丢入，自动识别类型 | 部分已有（content-ingestion.js），需扩展 |

**Feed 卡片规格**（复用 AI HOT 的信息结构）：
- AI HOT 已有的标签全部复用
- 摘要（AI HOT 提供的直接用）
- "全文解读"入口（触发 ② 理解阶段，按需执行）
- "跳转原文"按钮（永远保留）

### Source：优质源登记处（不是订阅系统）

用户丢入公众号名称 / X 链接 / 网页链接 → 自动识别身份 → 登记为信息源。登记后的效果**只有两个**：该源的内容进入 Feed，且高权重排序。

track_mode 分层（实现 ADR-007 的 active 档）：

| 情况 | track_mode | 行为 |
|------|-----------|------|
| AI HOT 已覆盖该 builder | passive | 纯标记 + Feed 加权，零成本 |
| 有 RSS 的博客/Newsletter/播客 | active-rss | RSS 轮询 |
| X / YouTube 且 AI HOT 未覆盖 | active-query | 轻量定期查询（优先经 skills 管道，浏览器抓取兜底） |
| 微信公众号 | link-only | 标记 + 等 AI HOT 推送 + 跳转原文，不抓取 |

飞轮闭环入口：任何内容详情页提供"**把作者加为信息源**"一键操作。

### ② 理解（Understand）

即 NotebookLM 式解读，复用现有 Mode 1（content-ingestion + ephemeral-chat + translation）。

关键升级：产物不是"一段摘要"，而是**结构化解读**，规格按"可被创作阶段直接引用"设计：
- 核心论点（分条）
- 论据 / 案例 / 数据
- 金句（可直接引用）
- 与知识库中已有观点的异同（知识层建成后启用）
- 英文内容直接产出中文结构化版；英文无字幕音视频 → 转写 → 中文结构化（ASR 后置，见路线图）

对话仍然不落库（保持轻、快、用完即走），但提供一等的沉淀出口（见 ③）。

### ③ 沉淀（Retain）

NotebookLM 模式：**用户选择"保存到笔记"，不是所有对话都存**。

保存的单元是**素材卡片（Note）**：
- 结构化摘录（来自解读产物或对话片段）✅ 确定
- 来源引用（回链到 Content，创作时可溯源）✅ 确定
- 立场批注（同意/反对/存疑）⏸️ 用户未想好，做成可选字段，默认不强制，暂不做 UI 强化

### ④ 洞察（Synthesize）—— 两个引擎

**引擎 A：热点 → 选题（节奏化简报，不实时刷新）**

基于用户自己的信息流做 Story 聚类（不做全网无差别挖掘），在聚合之上再走一步——产出**选题建议**，而非简单呈现聚合信息：

- **日报**：今日焦点 + 2-3 个选题建议。一个选题 = 切入角度 + 为什么是现在 + 支撑素材 + 共识/非共识点
- **周报**：本周动向 —— 哪些主题在升温/降温、知识库里哪些活页发生了演进
- **月报**：月度趋势 —— 主题成熟度变化、值得立项深研的方向

日报置顶于 Feed 页；周报/月报独立入口，由定时任务生成。

**引擎 B：活知识库（Karpathy LLM-wiki 式自生长）**

这是"死知识变活知识"的载体，核心机制是**同化（assimilation）**：

- Topic 重新定义为**活页（Living Page）**：一份由 AI 维护的活文档（当前认知综述 + 各方观点 + 共识/非共识），不是文件夹
- 当新内容被保存为素材、或命中已追踪的 Topic 时，触发同化：AI 将新素材并入相关活页 —— 补充新证据、修正旧论断、标记矛盾点，并写入一条 changelog
- **changelog 即演进时间线**：Topic 的演进（emerging → active → mature）从修订历史中自然涌现，无需手动维护
- **涌现**：周报/月报生成时，AI 回顾近期所有同化记录，提出新活页建议（新兴概念）、跨活页关联、矛盾预警、新选题 —— 这就是"AI 通过已有内容不断涌现新内容"

成本控制：同化只作用于**用户保存的素材和已追踪 Topic 命中的内容**，不对全量 Feed 执行。质量门槛由"用户选择保存"这个动作天然提供。

### ⑤ 创作（Create）

同一素材集（Notes + 活页），按平台分化输出模板：

| 模板 | 特征 |
|------|------|
| X thread / 短图文 | 钩子 + 分条 + 结尾引导，可从单篇解读直接快速生成（轻量高频） |
| 公众号长文 | 大纲 → 草稿，深度综述型 |
| 口播视频脚本 | 开场钩子、口语化、节奏与时长约束（抖音/视频号） |

每段可溯源到素材卡片 → 原始 Content。

### ⑥ 发布（Publish）

第一版只做导出/复制（Markdown / 各平台格式化文本），不做自动发布。未来演化为发布 Agent。

## 四、数据模型

保留 Content / Source，Topic 升级，新增 Note / Idea / Report / Draft：

| 实体 | 定义 | 关键字段增量 |
|------|------|-------------|
| Content | 统一内容模型（已有） | 复用 AI HOT 标签；interpretation（结构化解读，JSON） |
| Source | 优质源登记处（已有，升级） | track_mode: passive / active-rss / active-query / link-only；weight |
| Note 🆕 | 素材卡片 | excerpt（结构化摘录）、content_id（来源引用）、stance（可选）、topic_ids |
| Topic（升级） | 活页 Living Page | body（AI 维护的活文档）、perspectives、changelog[]（演进时间线） |
| Idea 🆕 | 选题 | angle、why_now、supporting_note_ids、consensus / non_consensus、status（建议/采纳/已创作） |
| Report 🆕 | 日报/周报/月报 | period_type、story_clusters、idea_ids、page_changes |
| Draft 🆕 | 稿件 | platform（thread/长文/脚本）、段落级 note 引用 |

## 五、模块契约（面向未来的 Agent/插件拆分线）

| 模块 | 输入 | 输出 | 未来形态 |
|------|------|------|---------|
| acquire | 各源适配器（含 skills 管道适配器） | Content[] | 采集 Agent / 数据源插件 |
| understand | Content / 任意链接 | 结构化解读 + 对话 | 解读 Agent |
| retain | 解读产物 / 对话片段 + 用户选择 | Note | — |
| knowledge | Note / Topic 命中内容 | 活页更新 + changelog | 知识库 Agent |
| insight | 近期 Content + changelog | Report + Idea | 热点发现 / 选题 Agent |
| create | Idea / Topic / Notes | Draft（分平台） | 内容生成 Agent |
| publish | Draft | 导出物 | 发布 Agent |

## 六、构建路线图

| 里程碑 | 内容 | 对应飞轮环节 |
|--------|------|-------------|
| **M1 沉淀层**（先做，已确认） | 素材卡片（保存到笔记流程）+ 来源引用；"加为信息源"闭环；Source 自动识别登记处；万能收口扩展（X 帖 / 公众号单篇跳转） | 解读→沉淀→追踪 |
| **M2 洞察层** | Story 聚类 + 日报（今日焦点 + 选题建议）；Feed 卡片改版（AI HOT 标签 + 摘要 + 全文解读 + 跳转）；GitHub Trending + skills 管道接入；单篇解读 → X thread 快速生成（轻量创作先行） | 发现→选题 |
| **M3 知识层** | Topic 活页 + 同化机制 + changelog 时间线；周报/月报（含涌现：新活页建议、关联、矛盾预警） | 活知识库 |
| **M4 创作层** | 公众号长文 + 口播脚本模板；段落级溯源；导出 | 创作→发布 |
| **M5 补强** | ASR（英文音频 → 中文结构化）；小宇宙接入；发布辅助 | — |

## 七、明确不做 / 后置

- ❌ 全网无差别趋势挖掘（只聚合自己的信息流）
- ❌ 公众号逆向抓取（单篇跳转原文即可）
- ❌ 重建订阅推送系统（Source 只是登记 + 加权）
- ❌ 实时刷新的热点（日/周/月节奏化简报）
- ❌ 自动发布（第一版导出即可）
- ⏸️ ASR / 小宇宙（重要但后置，M5）
- ⏸️ 立场批注 UI（字段预留，用户未确认需求）
- ⏸️ 移动端 / 多用户 / 协作

## 八、对既有决策（HANDOFF-2026-07-13 第四部分）的修订

| 原决策 | 结论 |
|--------|------|
| 决策 1：Source 是被动标记层 | **修正**：实现 ADR-007 的 active 档，做成"优质源登记处"（自动识别 + 加权），仍不是订阅系统 |
| 决策 2：公众号不抓取 | **维持**：单篇跳转原文 + AI HOT 摘要 |
| 决策 3：不做 Insight Mining | **重新画界**：全网挖掘维持不做；基于自己信息流的热点聚合 + 选题建议是**核心功能** |
| 决策 4：界面按模式分离 | **维持** |
| 决策 5：即兴分析基于原文 | **维持并升级**：原文 → 结构化解读，降级需显式标注 |
| Mode 1 无状态 | **维持**对话不落库，新增"保存为素材卡片"一等出口（NotebookLM 模式，用户选择保存） |
| Topic 是演进的研究线 | **升级**：Topic = AI 维护的活页，演进时间线从 changelog 涌现，替代手动归类为主的设计 |
