# Knowledge Workbench — 框线图集合（v1）

> 本文档汇总当前讨论确定的页面框线图，作为后续 HTML 原型和实现的依据。
> 配套架构文档：`SYNTHESIZED-ARCHITECTURE.md`、`DECISIONS.md`。
> 本文档记录的是**信息架构和交互结构**，不含视觉样式（配色/字体见 `prototype/` 下的可交互原型）。

---

## 核心设计原则（贯穿所有页面）

1. **Feed 默认是全量流**，不是"关注列表"。权威/新兴来源是排序信号和视觉标记，不是过滤条件。
2. **Source 是可信度标签，不是订阅关系**。没有独立的"Source 浏览"一级导航，Source 信息附着在内容卡片上。
3. **Topic 是研究和创作的主战场**。列表按"最近活跃度"排序，不按"演进阶段"分组（演进阶段降级为卡片标签）。
4. **近期焦点（基于 Story 聚类）** 是压缩信息密度的核心模块，在 Feed 主页（全局）和 Topic 详情页（scoped）各出现一次。
5. **搜索模糊主题** 走"先查已有内容库 → 不足则触发深度调研 → 用户确认后转为常驻 Topic"的路径,不是无脑生成。
6. **手动添加数据源** 需要体现 ADR-007 的成本分层（passive/active/link-only），不能许诺做不到的主动抓取。
7. **即兴分析必须基于原文，不能用摘要顶替**（2026-07-12 追加）：AI HOT 同步进来的内容只有平台给的短摘要（article 类型实测平均 241 字），选中分析时：
   - `tweet` 类型：摘要基本等价于原文全文（推文本身就短），且平台层面不做抓取（ADR-007），直接用摘要，不去抓 X 链接
   - `article`/`paper`/`repo` 类型：选中时实时抓取原文（复用 `content-ingestion.js` 的 `ingestUrl`），抓取失败或超时（15秒）则降级回摘要，且必须在材料里显式标注"无法获取原文，以下基于摘要"——不能悄悄降级掩盖这个事实。已在 `backend/src/services/ephemeral-chat.js` 的 `resolveContentBody()` 实现并验证。

## 5. 添加信源 · 方式 A：单个账号/频道

用于：即兴分析中发现的某个具体人/频道，用户想让系统持续追踪。
对应 ADR-006 的"涌现关注"路径 + ADR-007 的成本分层硬约束。

```
┌──────────────────────────────────────────────────────────────┐
│ 🔖 添加信源                                          [关闭 ×] │
│ ────────────────────────────────────────────────────────────  │
│ 粘贴链接或输入信息：                                          │
│ [https://x.com/karpathy 或 公众号名称...]                     │
│                                                                │
│ 检测到: X (Twitter) · @karpathy                               │
│                                                                │
│ 追踪方式：                                                     │
│  ⦿ ✅ 主动追踪（已配置 X API，实时同步新内容）                │
│  ○  ⚠️ 被动等待（不接 API，仅当此人内容出现在其他源时收录）  │
│                                                                │
│ ────────────────────────────────────────────────────────────  │
│ 若检测到的是公众号：                                          │
│  ○ 🔒 被动等待（固定选项，公众号不支持主动抓取）              │
│     不承诺主动更新，仅在 AI HOT 等源命中时收录 + 跳转原文阅读 │
│                                                                │
│ [确认添加]                                    [取消]          │
└──────────────────────────────────────────────────────────────┘
```

**要点**：
- 追踪方式的可选项由平台类型决定，不是用户随意选（ADR-007 硬约束）
- 公众号只有一个选项，且明确写出"不承诺主动更新"，避免用户误解产品能力
- 添加后这个 Source 出现在内容卡片的作者标签上，供后续排序/信任度使用,不产生独立的"关注列表"页面

## 待确认事项（供下一轮调研/决策参考）

1. **Story 聚类的实现粒度**：AI HOT 本身可能已做了部分聚类/去重,先确认它的现有能力边界，避免重复实现。当前决定：优先在 **Topic 层**做聚类（内容归类到主题），Story 层（同一事件多源识别）视工程成本决定是否 Phase 1 做,可先用"按评分/时间排序"占位近期焦点模块。
2. **信源池支持范围**：取决于另一份"Feed 信息源扩展"调研结论（Reddit / GitHub Trending / Hacker News / RSS 等哪些值得接入）。
3. **X/YouTube 主动追踪的 API 成本**：取决于同一份调研里"Twitter/X 内容流"章节的结论（官方 API 是否收费、额度多少）。
