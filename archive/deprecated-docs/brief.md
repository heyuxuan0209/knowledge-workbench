# Knowledge Workbench Brief

## 项目定位

这是一个面向个人知识管理与知识工作流的产品项目。

项目当前聚焦的不是做一个泛内容平台，也不是先做一个大而全的多功能系统，而是先解决我自己在 AI builder、AI 产品、AI 应用、可读技术精选 领域的信息处理闭环。

我希望这个项目逐步成为一个可持续使用的个人知识工作台，让我能够把外部高质量信息转化为自己的知识资产、内容素材和产品判断。

## 核心目标

当前要解决的核心链路是：

1. 高质量信息搜索与发现
2. 对英文或分散信息进行中文摘要和快速理解
3. 去重、筛噪，减少信息过载
4. 把值得保留的内容沉淀进自己的知识库
5. 基于沉淀内容生成选题、产品 idea、草稿或分享内容

目标不是单纯“收集更多信息”，而是建立一个真正服务于我个人输入、理解、沉淀、输出的知识工作流。

## 当前阶段理解

当前阶段的重点不是马上做完整平台，而是：

- 先把问题想清楚
- 先验证最小闭环是否真的对我有价值
- 先用少量高质量信息源跑通链路
- 先建立适合我的消费和沉淀方式

只有在最小闭环被验证后，才进入更完整的产品设计、架构设计和开发扩展。

## 当前推荐的 MVP 闭环

当前更合理的 MVP 主链路是：

抓取 -> 去重 -> 中文摘要 -> 人工判断 -> Obsidian 落盘

这条链路的含义是：

- 先从少量高质量信息源开始
- 对信息做基础清洗和去重
- 用 LLM 生成适合我阅读的中文摘要和价值判断
- 保留人工筛选，避免错误内容直接沉淀
- 只把真正值得保留的内容写入知识库

## 项目原则

1. 先验证价值，再扩张复杂度
2. 不从 0 造轮子，优先复用成熟工具
3. 信息源先少而精，不追求一开始全覆盖
4. Obsidian 是精选知识库，不是原始信息垃圾桶
5. 英文内容的中文摘要和价值判断是核心能力
6. 技术内容可以保留，但应以可读、可理解、可应用为前提
7. 系统的目标是提升我的判断和输出效率，而不是堆功能

## 当前推荐技术方向

当前阶段优先考虑的技术路线：

- 信息获取：RSS / RSSHub
- 存储：SQLite
- 服务层：Express
- 内容处理：LLM 中文摘要、翻译、价值判断
- 输出：Markdown 写入 Obsidian / LLM Wiki

这表示当前优先做轻量、稳定、便于快速迭代的实现，而不是一开始上复杂基础设施。

## 暂不优先做的内容

当前阶段不优先投入以下方向：

- 大而全的 Web 前端
- 复杂评分系统
- 复杂标签体系
- 多角色权限或账号体系
- n8n 等重工作流编排
- 自动邮件日报
- 自动多平台发布
- 大规模爬虫或高风险抓取链路
- 过早做完整产品化包装

这些内容不是永远不做，而是当前不应该进入 MVP 主路径。

## 信息源边界

当前项目关注的信息源重点包括：

- AI builder 实战内容
- AI 产品与工具更新
- AI 应用案例
- 可读的技术精选
- 方法论、工作流、产品判断

不以底层研究、纯 benchmark、纯 infra 更新为主，但如果某些技术内容能够被我看明白，并能转化为产品或工作流启发，也应该保留。

信息源体系需要可扩展，并允许后续从高命中核心源的关注链、引用链、互动链中继续发现新源。

## 与现有知识库的关系

这个项目不是替代我已有的 Obsidian / LLM Wiki，而是增强它。

它承担的作用是：

- 帮我更稳定地发现信息
- 帮我更高效地理解和筛选信息
- 帮我把高价值内容沉淀成结构化笔记
- 帮我把沉淀内容进一步转化为 idea、草稿和分享内容

## 当前输出的最小单位

现阶段更适合的单条内容输出格式应尽量简单，至少包括：

- 中文标题
- 来源
- 中文一句话摘要
- 为什么值得看
- 原文链接

如果后续验证有效，再逐步补充更复杂的结构化字段。

## 当前最重要的下一步

1. 基于少量高质量信息源整理第一版可执行清单
2. 明确单条内容的最小展示格式
3. 跑通从抓取到 Obsidian 落盘的最小闭环
4. 在实际使用中验证哪些源值得保留，哪些该淘汰

## 文档使用原则

- `brief.md` 用来记录这个项目的稳定目标、阶段共识和边界
- `handoff.md` 用来记录最近一轮讨论进展和下一步接力点
- `decisions.md` 用来记录已经确认的关键决策

当项目目标、MVP 边界、核心工作流或技术方向发生稳定变化时，应更新本文件。
