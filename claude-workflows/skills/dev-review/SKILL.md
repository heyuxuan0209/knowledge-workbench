# 开发-review 流程(dev-review)

## 三层验证,各管一层

| 层 | 谁 | 管什么 | 到不到用户面前 |
|----|-----|--------|----------------|
| 机器层 | 测试/lint/构建 | 细节正确性 | 不到(hook 强制) |
| Agent review 层 | reviewer subagent | 方向符合性(是否忠实于设计文档) | 不到,除非 S0/S1 |
| 用户层 | 用户 | 价值判断 | 只看三段式报告 |

## 流程

### Step 1: 读设计文档

读取指定的 `docs/designs/<主题>.md`。没有就停下,建议用户先走 kickoff-debate。
提取:验收判据(C)、决策(D)、已知风险。

### Step 2: 测试名先行(方向确认,唯一一次前置打扰)

把验收判据翻译成**测试名清单**(只有名字和一句话意图,不写实现):

```
- test_素材卡片保存后可通过content_id回溯来源 (对应 C1)
- test_未选择保存的对话不产生任何Note记录 (对应 C3)
- ...
```

发给用户看一眼:"这些是我打算验证的行为,对齐吗?"用户确认后再动手。
这一步是用户审方向的最快方式——审测试名就是审行为规格。

### Step 3: 实现

1. 先把 Step 2 的测试写出来(允许先红)。
2. 小步实现,直到测试全绿。
3. 测试必须锚定验收判据,禁止堆砌与判据无关的细节测试来刷绿色。
4. 完工前 hook 会强制跑测试门(.claude/hooks/test-gate.sh),不过不许收工。

### Step 4: 独立 review(优先 Codex,降级 subagent)

测试全绿后,按以下优先级选择审查者:

**路径 A(优先): Codex 跨模型审查**——若 codex-plugin-cc 已安装且 Codex CLI 可用,执行:

```
/codex:adversarial-review 对照 <设计文档路径> 审查本次改动:
1) diff 是否忠实实现了 <本次对应的 D 编号清单>?有无偷偷简化或偏离?
2) 有无设计文档未声明的架构偏移(新依赖/新表/新通信方式)?
3) 测试是否真的锚定验收判据(C 编号),有无 mock 短路关键路径、断言过弱?
禁止提: 命名、代码风格、lint/类型检查可捕获的问题、微优化、与本次 diff 无关的历史问题。
```

Codex 返回后,**必须做一轮验证**(不许照单全收):对照设计文档逐条判断其意见真伪,
把确认为真的映射为 S0/S1/S2(标准见下),误报明确标注"已核实为误报+理由"后丢弃。

**路径 B(降级): reviewer subagent**——Codex 插件未装、CLI 不可用或调用失败时,
用 Task 工具唤起 `reviewer` subagent,传给它:
- diff(git diff 或变更文件清单+内容)
- 设计文档路径
- 本次实现对应的决策编号清单

**不传**你的实现思路和对话历史,保持它上下文干净。

两条路径的严重度标准统一:
- S0: 违背设计决策或验收判据,合入即错
- S1: 未声明的架构偏移、判据绕过、触碰已知风险
- S2: 方向正确但值得记录的遗留项

### Step 5: 处置 review 意见

- S0/S1:必须修复,或给出理由标记 DISPUTED 留给用户裁决。
- S2:记入报告"遗留项",不阻塞。
- 修复后如 diff 变动大(新增文件/改动超过原 diff 的 1/3),重新过一遍 Step 4。
- 报告中注明本次走的是路径 A 还是 B。

**注意:不要启用 codex 插件的 review gate**(每次收工自动烧一次 Codex 审查,
易陷入拉锯循环且误报无人过滤)。测试门由 test-gate.sh 负责,Codex 审查只在本步手动触发一次。

### Step 6: 三段式报告(交给用户的唯一产物)

```markdown
# <功能> 完工报告

## 一、改了什么
(5 行以内的功能描述 + 变更文件清单)

## 二、对应设计文档哪些决策
- D1: 已实现,方式是 ...
- D3: 部分实现,偏离点见下
- (未覆盖的 D 编号明示"本次不涉及")

## 三、偏离与风险(只列 S0/S1)
- [S1][DISPUTED] reviewer 认为 X,我认为 Y,请你裁决
- (无则写"无")

## 附:验证过的行为
(Step 2 的测试名清单 + 全绿截图/输出摘要)
```

用户只需要:读报告 → 裁决 DISPUTED 项 → 决定合入与否。**不要求用户读代码。**

## 禁止事项

- 禁止跳过 Step 2 直接写代码。
- 禁止在测试未全绿时触发 review(浪费一次干净上下文/一次 Codex 额度)。
- 禁止未经验证直接把 Codex 意见塞进报告(必须先核真伪)。
- 禁止把 reviewer 的 S2 意见混入报告第三节稀释注意力。

## 复盘钩子(棘轮)

完工后问用户:"这轮里你在哪些地方不得不人工纠偏?"
纠偏点 → 追加为本 SKILL.md 或 reviewer.md 的新规则。
连续 3 次零纠偏的步骤,才考虑进一步自动化(如自动触发本流程)。
