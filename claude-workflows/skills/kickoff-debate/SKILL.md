# 冷启动收敛流程(kickoff-debate)

## 目标与终点

本流程的终点**不是"讨论完了"**,而是产出一份设计文档 `docs/designs/<主题>.md`,满足:
未决问题清单归零(每个问题都有归宿),需要用户拍板的决策全部已拍板。

你(主 agent)扮演 **Builder**。挑战工作必须交给 `challenger` subagent(独立上下文),
不允许自己左右互搏——同一个上下文里的自我批评是无效的。

## 铁律

1. **Phase 0 完成前,禁止输出任何方案、架构、技术选型。** 哪怕用户直接问"用什么技术栈",也先回到问题澄清。
2. "达成一致"的唯一定义 = 未决问题清单归零。语气客气、双方不再反驳,都不算收敛。
3. 价值权衡类问题(快 vs 稳、简单 vs 完备、成本 vs 体验)**永远不由 agent 裁决**,必须进"待用户拍板"清单。
4. 对抗最多 3 轮。轮数用完仍未解决的问题,打包成决策清单交用户,流程照样结束——不许拖第 4 轮。

## Phase 0:问题澄清(禁止出方案)

1. 用自己的话复述:「我理解你要解决的问题是 X,关键约束是 Y,我最不确定的是 Z」。
2. 一次最多问 3 个问题,优先问"为什么现在做""不做会怎样""谁在什么场景用"。
3. 迭代直到用户确认问题陈述没跑偏。
4. 和用户一起写下**验收判据**:好方案必须满足的 5-7 条,每条可检验。由用户逐条确认。

Phase 0 的产出(写入设计文档草稿):

```markdown
## 问题陈述
(一段话,用户已确认)

## 验收判据(用户已确认)
- C1: ...
- C2: ...
```

**门槛:用户明确说"问题陈述和判据 OK"之后,才能进入 Phase 1。**

## Phase 1:Builder 出方案

方案必须按以下格式写(自由散文无效):

```markdown
## 方案(v1)

### 决策
- D1: (一条具体的、可被攻击的决策)
- D2: ...

### 假设清单(自报)
- A1: (本方案成立所依赖的假设)
- A2: ...

### 薄弱点(自报)
- W1: (我自己最没把握的地方)
```

自报假设和薄弱点是强制的,写"无"视为流程违规。

## Phase 2:Challenger 攻击(subagent)

用 Task 工具唤起 `challenger` subagent,**只传给它**:问题陈述、验收判据、方案全文。
不传你的推理过程、不传对话历史——它的价值就在于上下文干净。

它会返回编号 issue 清单(格式见 challenger 定义)。

## Phase 3:Builder 逐条处置

对每条 issue,只有四种出路,逐条明示:

| 处置 | 含义 | 去向 |
|------|------|------|
| RESOLVED | 论证解决,给出理由或修改方案 | 关闭,方案升版(v2) |
| NEEDS-EVIDENCE | 讲理定不了,需要 spike/查资料 | 进「待验证」清单,写明验证方法 |
| USER-DECISION | 纯价值权衡 | 进「待用户拍板」清单,附选项+各自代价 |
| ACCEPTED-RISK | 承认但接受 | 进「已知风险」清单 |

修改后的方案(v2)连同处置表,再送 challenger 一轮。最多 3 轮。

## Phase 4:收口

1. 把「待用户拍板」清单一次性交给用户,每项格式:
   `Q1: (问题) | 选项A: ...(代价) | 选项B: ...(代价) | Builder 倾向: A,因为...`
2. 用户拍板后,把结论写为 D 编号决策,合入设计文档。
3. 最终设计文档结构:

```markdown
# <主题> 设计文档
状态: 已收敛 | 日期: <当天> | 轮数: N

## 问题陈述
## 验收判据
## 决策(D1..Dn,含用户拍板项,标注来源)
## 假设清单(存活的)
## 待验证清单(NEEDS-EVIDENCE 项 + 验证方法)
## 已知风险(ACCEPTED-RISK 项)
## 对抗记录摘要(每轮 issue 数、关闭数)
```

4. 保存到 `docs/designs/<主题>.md`,告诉用户:这份文档是后续 dev-review 流程的输入。

## 复盘钩子(棘轮)

流程结束后问用户一句:"这轮里你有哪些地方觉得流程不对、不得不人工纠偏?"
如有,建议把纠偏点作为新规则追加进本 SKILL.md。连续 3 轮零纠偏的步骤才考虑进一步自动化。
