## 第一部分：这个产品要解决什么问题

### 用户画像

个人研究者 / 知识工作者 / Builder：
- 每天消费大量碎片信息（YouTube 视频、技术博客、Twitter、论文）
- 想从信息中提取洞察，但没时间全看
- 需要追踪某个主题（如"Agent 架构"）的演进脉络
- 积累了很多素材，但创作时找不回来

### 真实场景（用户视角）

**场景 1: 即兴分析**
> "我在 YouTube 刷到 Andrej Karpathy 的 40 分钟视频讲 LLM，没时间全看，但想快速了解核心观点。"

传统方案：
- 硬着头皮看完（耗时）
- 看评论区（不可靠）
- 找 AI 总结（但 YouTube 字幕需要手动复制）

**场景 2: 主题追踪**
> "我在研究'Multi-Agent 系统'，想知道这个领域从 Hype → 批判 → 最佳实践的演进过程，各家观点对比。"

传统方案：
- Notion 手动归档（被动，无法看到演进）
- RSS 阅读器（只是推送，没有主题聚合）
- Perplexity 一次性搜索（无法持续追踪）

**场景 3: 创作桥接**
> "我积累了 50 篇关于 RAG 的文章/视频，想写一篇综述，但不记得哪篇说了什么。"

传统方案：
- 重新翻一遍（耗时）
- ChatGPT 上传文件（上下文有限，无法追溯来源）

### 核心痛点

1. **信息过载** → 没时间全看，但又怕错过重要信息
2. **无法沉淀** → 看完即忘，信息没有变成可复用的知识
3. **演进脉络缺失** → 只能看到"现在的观点"，看不到"为什么变成这样"
4. **素材追溯困难** → 创作时想引用，找不回之前看的内容

## 第二部分：产品方案（我们的解法）

### 一句话定位

**从信息消费到知识产出的个人研究工作台**  
把每天刷到的碎片信息，沉淀为可追踪的信息网络、可演进的研究主题、可创作的素材资产。

### 核心差异化

| 对比维度 | 传统工具 | Knowledge Workbench |
|---------|---------|---------------------|
| AI HOT / Hacker News | 推送资讯 | 资讯 + **即兴分析** |
| Notion / Obsidian | 被动存储 | 主动演进（Topic Timeline） |
| NotebookLM | 单次对话 | 持续追踪（Source + Topic） |
| Perplexity | 一次性搜索 | 长期积累 + 演进 |

**不是重复造轮子**：资讯聚合层复用现成的（AI HOT / RSS），差异化在**聚合之上的这一层**。

### 三种使用模式（心智模型）

这不是三个并列功能，是三种不同的**心智状态**：

#### Mode 1: 即兴分析 (Ephemeral)
- **心智**: 轻、快、用完即走
- **触发**: 刷到内容时的"快速看懂"需求
- **交互**: 选中内容 → 弹窗对话 → 问答 → 可选保存
- **特点**: 无状态（不落库），对话历史随关闭即弃

#### Mode 2: 主题研究 (Persistent)
- **心智**: 慢、深、长期演进
- **触发**: "我要持续追踪某个主题"
- **交互**: Topic 为中心 → 内容自动/手动归类 → 演进时间线 → perspectives 对比
- **特点**: 有状态，Topic 会随时间演进

#### Mode 3: 创作桥接 (Output)
- **心智**: 产出导向
- **触发**: "我要基于积累的素材写文章"
- **交互**: 选择 Topic → AI 生成大纲 → 草稿编辑 → 导出（飞书/Notion/Markdown）
- **特点**: 每段可回溯引用来源

### 核心实体（数据模型）

#### Content（内容）
- 统一模型：article / video / tweet / paper / repo
- 多语言一等公民：zh_title / en_title / zh_body / en_body
- 状态追踪：fetch_status (pending / success / failed)
- 来源：AI HOT / HackerNews / RSS / 手动粘贴

#### Source（身份标记层）
- **不是订阅系统**，是轻量标记层
- 作用：可信度信号（权威来源 vs 新兴来源）
- track_mode 分层（ADR-007）：
  - passive: AI HOT 已覆盖，纯被动推送（零成本）
  - active: X/YouTube 未覆盖，主动查询 API（有成本）
  - link-only: 微信公众号（不抓取，只跳转）

#### Topic（研究主题）
- **不是文件夹**，是会演进的研究线
- evolution: 记录主题的演进阶段（emerging → active → mature）
- perspectives: 不同 Source 的观点对比
- 排序：按"最近活跃度"，不按阶段分组

## 第四部分：关键决策（必须理解的硬约束）

### 决策 1: Source 不是抓取系统

**错误理解**: "关注一个 YouTube 频道 → 产品主动抓取他的视频"  
**正确理解**: "AI HOT 推送了他的视频 → 产品标记这是谁 → Feed 排序加权"

**为什么这样设计**:
- 重建 YouTube 订阅推送是巨坑（API 配额、实时性、成本）
- Source 的价值不在"抓取"，在"可信度标记"（权威来源 vs 新兴来源）
- 对标 X 的推荐机制：先从内容里发现感兴趣的人 → 关注 → 未来加权

**成本分层（ADR-007）**:
| 平台 | track_mode | 成本 |
|------|-----------|------|
| AI HOT 已覆盖 | passive | 零（等推送） |
| X / YouTube 未覆盖 | active | 有（主动查 API） |
| 微信公众号 | link-only | 不做（无公开 API） |

### 决策 2: 微信公众号不做抓取

**为什么**:
- 无公开 API，逆向抓取合规风险高
- 公众号本身不稳定（防逆向措施频繁变化）

**怎么做**:
- 只被动等 AI HOT 推送
- 提供"跳转原文"按钮，不承诺全文展示

### 决策 3: 不实现 Insight Mining

**什么是 Insight Mining**:
- "自动挖掘产品机会"
- "AI 主动推送趋势冲突"
- "系统告诉你应该关心什么"

**为什么不做**:
- 这是**伪需求**，未经验证
- 用户要的是"我研究 X 主题时的洞察"，不是"系统告诉我该研究什么"
- 洞察应该是 Mode 3 创作对话的**自然产出**，不是独立功能

### 决策 4: 界面按使用模式分离

**旧版失败教训**:
- 把资讯流（快速扫描）和工作区（深度对话）塞进同一屏幕
- 结果：两种都做不好（扫描时工作区占地方，深度工作时 Feed 干扰注意力）

**新架构**:
- Mode 1/2/3 独立界面，不强塞
- Mode 1 用"弹窗"而不是"固定栏"（用完即走）

### 决策 5: 即兴分析必须基于原文

**错误做法**: 用 AI HOT 给的 241 字摘要顶替原文

**正确做法**:
- tweet 类型：摘要≈原文（推文本身就短），直接用
- article/paper/repo：选中时实时抓取原文（复用 content-ingestion.js）
- 抓取失败/超时（15秒）：降级回摘要，**但必须显式标注"无法获取原文，以下基于摘要"**

**为什么**:
- 241 字摘要无法支撑深度对话
- 不能悄悄降级（用户会以为 AI 读过全文，实际没有）

## 第五部分：已踩过的坑（技术层面）

### 坑 1: Deepseek 上下文注入方式

**问题**: 把材料/背景信息用 `system` role 传 → Deepseek 会忽略

**原因**: Deepseek 在某些情况下无法正确识别 system 消息（具体原因待查，可能是历史消息干扰）

**解决**: 把背景信息拼接进当前这条 `user` 消息的前缀
```javascript
const materialsBlock = `# 参考材料\n\n${materials.join('\n\n')}\n\n请基于以上材料回答问题。`
messages.push({ role: 'user', content: materialsBlock + userQuestion })
```

**位置**: `backend/src/services/ephemeral-chat.js`

### 坑 2: youtube-transcript 代理配置

**问题**: 设置 `HTTP_PROXY` / `HTTPS_PROXY` 环境变量 → youtube-transcript 仍然无法访问

**原因**: Node.js 内置 fetch (undici) 不读这套环境变量约定

**解决**: 显式传 `undici` 的 `ProxyAgent`
```javascript
import { ProxyAgent } from 'undici'
const proxyUrl = process.env.YOUTUBE_PROXY_URL
const fetch = proxyUrl 
  ? (url, opts) => fetch(url, { ...opts, dispatcher: new ProxyAgent(proxyUrl) })
  : fetch
YoutubeTranscript.fetchTranscript(videoId, { fetch })
```

**排查方法**: 先用 `curl -x <proxy> <url>` 验证代理本身可用，再检查 HTTP 客户端库是否支持

**位置**: `backend/src/services/content-ingestion.js`

### 坑 3: SQLite 字段类型约束

**问题**: `has_translation: false` → INSERT 报错 "cannot be bound to parameter 7"

**原因**: SQLite 用 INTEGER 存布尔值（0/1），JavaScript 的 `true/false` 无法直接绑定

**解决**: 手动转换
```javascript
content.has_translation = 0  // 不能用 false
```

**位置**: `backend/src/services/rss.js` (已修复)

### 坑 4: CHECK 约束陷阱

**问题**: `input_method: 'feed_auto'` → 报错 "CHECK constraint failed"

**原因**: `schema-v3.sql` 有严格的枚举约束
```sql
CHECK (input_method IN ('feed', 'url_auto', 'url_manual', 'text_paste', 'file_upload'))
```

**解决**: 查 schema，用正确的枚举值
```javascript
content.input_method = 'feed'  // 不能用 'feed_auto'
content.source_app = 'rss'     // 不能用 'rss_少数派'
```

### 坑 5: 数据库用 node:sqlite，不要用 better-sqlite3

**问题**: `npm install better-sqlite3` → node-gyp 编译失败

**原因**: Node 26 上 better-sqlite3 编译有问题

**解决**: 用 Node.js 26 内置的 `node:sqlite`
```javascript
import { DatabaseSync } from 'node:sqlite'
```

## 附录：常见问题

**Q: 为什么不用向量数据库？**  
A: Phase 1/2 不需要。Topic 聚类用轻量算法（TF-IDF + 相似度）够用。Phase 3 如果要做"发现关联"，再考虑 Embedding。

**Q: 为什么用 Deepseek 不用 Claude？**  
A: 成本。Deepseek ¥1/M tokens，Claude ¥60/M tokens。Phase 1 高频调用（摘要/翻译/对话），成本敏感。

**Q: 为什么不做移动端？**  
A: Phase 1-3 聚焦桌面端"深度工作"场景。移动端是"碎片消费"，优先级低。

**Q: 为什么数据库用 SQLite 不用 PostgreSQL？**  
A: 个人工具，单机部署够用。未来如果要做协作/云端，再迁移。
